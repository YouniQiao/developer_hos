## 问题现象

在隐式Want中，需要使用pathRegex正则匹配文件类型，使用下面的配置后匹配失败，无法拉起应用，提示“暂无可用打开方式”，如下图：

```
"skills": [
  {
    "actions": [
      "ohos.want.action.viewData"
    ],
    "uris": [
      {
        "scheme": "file",
        "host": "file",
        "port": "21",
        "pathRegex": ".*txt",
        "linkFeature": "fileopen"
      }
    ]
  }
]
```

![](./img/631d944e.png "点击放大")

## 背景知识

* [Want](https://developer.huawei.com/consumer/cn/doc/harmonyos-guides/want-overview)是一种对象，用于在应用组件之间传递信息。根据启动目标应用组件时传入参数不同区分为显示Want和隐式Want：
  + 显式Want：在启动目标应用组件时，调用方传入的Want参数中指定了abilityName和bundleName，称为显式Want。
  + 隐式Want：在启动目标应用组件时，调用方传入的Want参数中未指定abilityName，称为隐式Want。
* 按照[隐式Want匹配原理](https://developer.huawei.com/consumer/cn/doc/harmonyos-guides/explicit-implicit-want-mappings#隐式want匹配原理)规则匹配成功后会提示拉起应用，基本匹配规则如下：
  + 系统将调用方传入的Want参数（包含action、entities、uri、type和parameters属性）与已安装待匹配应用组件的skills配置（包含actions、entities、uris和type属性）进行匹配。
  + 当Want参数五个属性匹配均未配置，则隐式匹配失败。
  + 当parameters中的linkFeature字段取值不为空时，系统将优先进行linkFeature匹配。
* [skills标签](https://developer.huawei.com/consumer/cn/doc/harmonyos-guides/module-configuration-file#skills标签)标识UIAbility组件或者ExtensionAbility组件能够接收的Want的特征。uris标签中pathRegex标识uri与Want中的路径部分允许正则匹配。

## 问题定位

根据匹配规则，如果匹配失败，需要检查隐式Want的5个参数与skill配置项是否符合规则，skill的配置项的uris标签：

```
- actions：ohos.want.action.viewData
- scheme：file
- host：file
- port：21
- pathRegex：.*txt
- linkFeature：fileopen
```

可以看出期望匹配的uri类型是file://file:21/.\*txt，因此怀疑是Want入参的uri与期望匹配的uri类型不一致。

## 分析结论

配置项中pathRegex值错误，导致正则表达式无法匹配文件格式，拉起应用失败。

## 修改建议

修改skills配置项中的"pathRegex": ".\*txt"，变成"pathRegex": "\*.txt"。然后使用正确的uri验证修改后的skills配置项，以txt格式为例：

uri: "file://file:21/student.txt"

## 总结

隐式Want匹配关注五个核心参数，如果匹配不上，有可能是某个参数的配置不正确，逐个排查配置中的actions、scheme、host、port、path（pathRegex或pathStart）。

## 常见FAQ

Q：以下配置是否可以匹配test://openlink？

```
"uris": [
  {
    "scheme": "test",
    "host": "*"
  }
]
```

A：当配置文件只配置scheme时，可以仅匹配scheme，但如果配置了host，也会匹配host，且host不支持正则匹配，即\*只能匹配test://\*。
